FAQ

자주 묻는 질문

A simple site migration can take 1–3 hours, while a complex stack (MySQL, Docker, multiple apps) may take a full day. The biggest variable is database size — a 50GB database transfer can take 2–4 hours over a 1 Gbps link. Plan your migration during low-traffic hours.
단순한 사이트 이전은 1~3시간, MySQL·Docker·여러 앱이 포함된 복잡한 스택은 하루가 걸릴 수 있습니다. 가장 큰 변수는 데이터베이스 크기입니다. 50GB DB는 1Gbps 링크에서도 2~4시간이 걸릴 수 있습니다. 트래픽이 적은 시간대에 이전을 계획하세요.
Yes — do this 24–48 hours before your migration. A lower TTL means DNS changes propagate faster when you update the A record to point to your new server. After migration is stable (48h), restore TTL to 3600 or higher to reduce DNS query load.
네, 이전 24~48시간 전에 설정하세요. TTL이 낮으면 새 서버를 가리키도록 A 레코드를 변경했을 때 DNS 변경이 더 빨리 전파됩니다. 이전이 안정화(48시간 후)되면 TTL을 3600 이상으로 복원해 DNS 쿼리 부하를 줄이세요.
Use mysqldump --all-databases --single-transaction > backup.sql for InnoDB tables (no locking). Transfer via rsync or scp over SSH, then restore with mysql < backup.sql. For large databases (>10GB), consider using Percona XtraBackup for a hot backup with less downtime.
InnoDB 테이블은 mysqldump --all-databases --single-transaction > backup.sql을 사용하세요(잠금 없음). SSH를 통해 rsync 또는 scp로 전송하고, mysql < backup.sql로 복원하세요. 10GB 이상의 대용량 데이터베이스는 다운타임을 줄이기 위해 Percona XtraBackup을 고려하세요.
Yes — Let's Encrypt certificates are domain-validated. Install Certbot on the new server, then run certbot certonly --nginx -d yourdomain.com after DNS has propagated to the new IP. Do not copy the old certs; get fresh ones so the new server's IP is properly associated.
네, Let's Encrypt 인증서는 도메인 검증 방식입니다. 새 서버에 Certbot을 설치하고, DNS가 새 IP로 전파된 후 certbot certonly --nginx -d yourdomain.com을 실행하세요. 기존 인증서를 복사하지 말고, 새 인증서를 발급받아 새 서버 IP와 올바르게 연결하세요.
Wait at least 48 hours after DNS cutover before deleting the old server. This covers users still hitting the old IP due to cached DNS entries. Monitor your new server's access logs — when traffic to the old server drops to near zero, it's safe to decommission it.
DNS 전환 후 최소 48시간을 기다린 후 구 서버를 삭제하세요. 이는 캐시된 DNS 항목으로 인해 구 IP로 접속하는 사용자를 커버합니다. 새 서버의 접속 로그를 모니터링하다가 구 서버 트래픽이 거의 0에 가까워지면 삭제해도 됩니다.
